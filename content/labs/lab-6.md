## 1. Тема, мета, посилання

### 1.1 Тема
«Документування API за допомогою Swagger. Деплой Node.js-додатку. Підсумковий проєкт: REST API з MySQL».

### 1.2 Мета
Інтегрувати в `TaskFlow` документацію `Swagger/OpenAPI`, забезпечити перевірку маршрутів через `Swagger UI` та підготувати серверний застосунок до подальшого розгортання в production-середовищі.

### 1.3 Посилання
- Репозиторій власного проєкту (GitHub): буде додано після публікації.
- Розгорнутий застосунок: буде додано після розгортання.
- Репозиторій звітного HTML-документа (GitHub): буде додано після публікації.
- Звітний HTML-документ: буде додано після розгортання.

## 2. Хід виконання роботи

### 2.1 Загальна логіка завершального етапу
На завершальному етапі розвитку `TaskFlow` основну увагу було зосереджено на тому, щоб зробити API не лише функціональним, а й зручним для подальшої інтеграції та перевірки. Для цього було виконано два основні напрями робіт:
- документування маршрутів через `Swagger/OpenAPI`;
- підготовка застосунку до подальшого розгортання.

Оскільки на попередніх етапах уже було реалізовано:
- авторизацію;
- роботу із задачами та групами задач;
- upload вкладень;
- моніторинг стану сервісу;
- кешування та тестування,

додавання інтерактивної документації стало логічним фінальним етапом оформлення проєкту.

### 2.2 Інтеграція Swagger у проєкт
Для побудови документації було використано бібліотеки:
- `swagger-jsdoc`
- `swagger-ui-express`

На їх основі сформовано окремий конфігураційний файл `config/swagger.js`, у якому задано базову інформацію про API.

Фрагмент конфігурації:

```js
const definition = {
  openapi: "3.0.0",
  info: {
    title: "TaskFlow API",
    version: "1.0.0",
    description: "API documentation for TaskFlow"
  },
  servers: [
    {
      url: "http://localhost:3001",
      description: "Local server"
    }
  ]
};
```

У цьому блоці задаються:
- версія стандарту `OpenAPI`;
- назва сервісу;
- версія документації;
- локальна адреса сервера для тестування запитів.

### 2.3 Опис схем даних і захищеного доступу
У специфікації окремо було задокументовано механізм `JWT`-авторизації та базові структури даних, з якими працює сервіс.

Фрагмент секції `components`:

```js
components: {
  securitySchemes: {
    bearerAuth: {
      type: "http",
      scheme: "bearer",
      bearerFormat: "JWT"
    }
  },
  schemas: {
    LoginRequest: {
      type: "object",
      required: ["email", "password"]
    }
  }
}
```

Завдяки цьому в `Swagger UI` можна чітко побачити:
- які маршрути є захищеними;
- які поля необхідно передавати;
- які типи даних використовуються для задач, груп задач і авторизації.

### 2.4 Документування основних маршрутів
У документації були описані ключові endpoint-и сервісу:
- `GET /api/health`
- `GET /api/status`
- `POST /api/auth/login`
- `GET /api/task-groups`
- `POST /api/task-groups`
- `GET /api/tasks`
- `POST /api/tasks`
- `POST /api/uploads/attachments`

Приклад опису маршруту логіну:

```js
"/api/auth/login": {
  post: {
    summary: "Authenticate user",
    requestBody: {
      required: true,
      content: {
        "application/json": {
          schema: {
            $ref: "#/components/schemas/LoginRequest"
          }
        }
      }
    }
  }
}
```

Приклад опису маршруту створення задачі:

```js
"/api/tasks": {
  post: {
    summary: "Create task",
    security: [{ bearerAuth: [] }],
    requestBody: {
      required: true,
      content: {
        "application/json": {
          schema: {
            $ref: "#/components/schemas/TaskInput"
          }
        }
      }
    }
  }
}
```

Такий підхід дозволив відобразити в документації не лише службові маршрути, а й основну бізнес-логіку застосунку.

### 2.5 Документування маршруту завантаження вкладень
Оскільки в `TaskFlow` реалізовано додавання вкладень до задач, цей сценарій також було внесено до документації.

Фрагмент опису:

```js
"/api/uploads/attachments": {
  post: {
    summary: "Upload task attachment",
    security: [{ bearerAuth: [] }],
    requestBody: {
      required: true,
      content: {
        "multipart/form-data": {
          schema: {
            type: "object",
            properties: {
              file: {
                type: "string",
                format: "binary"
              }
            }
          }
        }
      }
    }
  }
}
```

Завдяки цьому upload-маршрут також став доступним для перевірки через веб-інтерфейс документації.

### 2.6 Підключення Swagger UI
Сформовану OpenAPI-специфікацію було підключено до Express-застосунку через окремий маршрут `/api-docs`.

Фрагмент інтеграції:

```js
const swaggerUi = require("swagger-ui-express");
const swaggerSpec = require("./config/swagger");

app.use("/api-docs", swaggerUi.serve, swaggerUi.setup(swaggerSpec));
```

Після запуску сервера сторінка документації стає доступною за адресою:

```text
http://localhost:3001/api-docs/
```

Ця сторінка дозволяє:
- переглядати всі основні маршрути;
- відкривати опис кожного endpoint-а;
- бачити структури вхідних даних;
- одразу виконувати тестові запити.

### 2.7 Підготовка до деплою
Для стандартизації запуску серверної частини було додано `Dockerfile`, який описує контейнеризований спосіб розгортання застосунку.

Фрагмент `Dockerfile`:

```dockerfile
FROM node:20-alpine

WORKDIR /app

COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile --prod

COPY . .

ENV NODE_ENV=production
EXPOSE 3000

CMD ["pnpm", "start"]
```

Крім цього, створено `.dockerignore`, щоб до контейнера не потрапляли службові та локальні файли:

```text
node_modules
logs
uploads
.env
.git
.DS_Store
coverage
```

У результаті застосунок отримав стандартизовану основу для подальшого запуску на типовій платформі розгортання.

### 2.8 Практичний результат
У підсумку сервіс `TaskFlow` отримав:
- інтегровану `Swagger/OpenAPI`-документацію;
- окрему сторінку `Swagger UI`;
- опис основних маршрутів, структур даних і захищених endpoint-ів;
- документацію upload-сценарію для вкладень;
- production-орієнтований `Dockerfile`;
- базову готовність до подальшого deployment-процесу.

Отже, серверна частина проєкту була оформлена у вигляді більш завершеного API-рішення, придатного для подальшої інтеграції та публічного запуску.
